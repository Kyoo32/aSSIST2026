## Multi-Turn Chatbot And SQL Guardrails

### Source

- [AI 기반 SQL 분석 에이전트 구축 - 11. Multiturn Chatbot](https://siapapa.github.io/day2/11-multiturn-chatbot/)

### 1. 단발 질의 vs 멀티턴 대화

- **단발 질의 (Stateless)** 는 한 번 묻고 한 번 답하는 구조다.
- 이전 질문을 기억하지 않기 때문에, 다음 질문이 앞 문맥을 참조하면 이해하지 못한다.

예:

- 사용자: `"남성 환자 수는?"`
- 봇: `"남성 환자는 15명입니다."`
- 끝

- **멀티턴 대화 (Stateful)** 는 이전 대화 문맥을 이어서 이해한다.

예:

- 사용자: `"남성 환자 수는?"`
- 봇: `"남성 환자는 15명입니다."`
- 사용자: `"그 중에 40세 이상은?"`
- 봇: `"남성 환자 중 40세 이상은 7명입니다."`

여기서 `"그 중"` 같은 표현은 이전 문맥이 있어야만 해석 가능하다.

> 한 줄 정리: 멀티턴 챗봇의 핵심은 "이전 대화를 기억해서 다음 질문을 해석하는 것"이다.

### 2. ChatState 데이터클래스

강의에서는 `ChatState`를 **대화 메모장**처럼 사용한다.

핵심 역할:

- `history`: 사용자/시스템 대화 이력 저장
- `last_sql`: 마지막 실행 SQL 저장
- `last_result`: 마지막 결과 저장

왜 필요한가:

- LLM은 원래 stateless다.
- 그래서 우리가 최근 대화와 이전 결과를 구조적으로 저장해 두고,
- 필요할 때 다시 프롬프트에 넣어줘야 멀티턴이 가능해진다.

즉, `ChatState`는 "기억이 없는 모델" 위에 얹는 **외부 기억 장치**다.

### 3. SQL 보안 가드레일

강의의 SQL 가드레일은 위험 SQL을 사전에 차단하려는 1차 방어선이다.

대표적으로 막으려는 위협:

- **DML 공격**: `DELETE`, `DROP`, `UPDATE` 같은 데이터 변경/삭제
- **SQL 인젝션**: 악의적 SQL 구문 삽입
- **테이블 무단 접근**: 허용되지 않은 테이블 조회
- **대량 조회**: `SELECT * FROM 큰_테이블` 같은 과부하성 쿼리
- **시스템 정보 노출**: `information_schema`, `pg_roles` 등 접근

강의에서 제시한 기본 방어 아이디어:

1. 위험 키워드 차단
2. 시스템 테이블/패턴 차단
3. 허용 테이블 화이트리스트
4. `LIMIT` 자동 주입 등 조회량 제한

> 한 줄 정리: SQL 가드레일은 "LLM이 만든 SQL을 그대로 믿지 말고, 실행 전에 최소한 한 번 더 검사하자"는 장치다.

### 4. 정규식만으로는 완벽하지 않다 — 보안 한계 설명

강의에서 강조하는 핵심은 이것이다.

> **정규식 기반 차단은 1차 방어선일 뿐이다.**

이유는 우회 가능성이 있기 때문이다.

예시:

- 대소문자 혼합: `DrOp TaBlE`
- 전각 문자: `ＤＲＯＰ`
- `CHR()` 함수 조합
- 동적 SQL
- 주석 우회: `DR/**/OP TABLE`

즉, 정규식은 일부 패턴은 막을 수 있어도, 모든 변형을 완벽하게 막지는 못한다.

그래서 프로덕션에서는 추가 보안 레이어가 필요하다.

- 데이터베이스 권한 제한
- SQL 파싱 기반 검증
- 화이트리스트 정책
- 실행 결과/로그 모니터링

### 한 줄 정리

- 멀티턴 챗봇은 `ChatState` 같은 상태 관리가 핵심이고, SQL 보안은 정규식 하나로 해결되지 않기 때문에 여러 방어층을 함께 가져가야 한다.
