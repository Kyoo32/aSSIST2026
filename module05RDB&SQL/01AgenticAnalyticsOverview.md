### 데이터 분석의 3세대

| 세대 | 방식 | 식당 비유 | 한계 |
| --- | --- | --- | --- |
| **1세대: BI 대시보드** | 분석가가 미리 만든 보고서 | 정해진 세트 메뉴 | 새로운 질문은 어렵다 |
| **2세대: Text-to-SQL** | 자연어 질문을 SQL로 1회 변환 | 메뉴판 보고 단품 주문 | 복잡한 질문과 오류 대응이 약하다 |
| **3세대: Agentic Analytics** | AI가 계획-실행-검증-재시도를 반복 | "맛있는 거 해주세요"처럼 목표 중심 요청 | 이번 학습의 핵심 대상 |

- 1세대는 정해진 분석만 볼 수 있다.
- 2세대는 자연어 질의가 가능하지만, 한 번의 SQL 생성에 크게 의존한다.
- 3세대는 질문을 분해하고, 중간 결과를 확인하며, 실패 시 다시 시도한다.

### 왜 "에이전트"인가?

- Text-to-SQL은 보통 **SQL 1회 생성 -> 실행 -> 종료** 흐름에 가깝다.
- 그래서 다음 문제가 생기기 쉽다.
  - 잘못된 SQL을 스스로 감지하지 못한다.
  - 복잡한 질문을 여러 단계로 나누지 못한다.
  - 결과가 이상해도 재시도하지 않는다.

- 에이전트는 이 한계를 넘기 위해 다음 흐름으로 동작한다.
  1. 질문 분석
  2. SQL 생성
  3. SQL 실행
  4. 결과 검증
  5. 필요 시 SQL 수정 후 재실행
  6. 최종 자연어 답변 생성

> 핵심 포인트: 에이전트는 "한 번 답하는 모델"이 아니라, "끝까지 문제를 해결하려고 반복하는 시스템"이다.

### 에이전트의 4가지 핵심 요소

1. **State (기억력)**
   - 이전 대화와 현재 작업 상태를 기억한다.
   - 예: 대화 이력, 현재 SQL, 중간 결과

2. **Tools (도구함)**
   - 에이전트가 실제로 사용할 수 있는 외부 기능이다.
   - 예: SQL 실행기, 검색 도구, AI 호출

3. **Planning (계획력)**
   - 복잡한 질문을 단계별로 분해한다.
   - 어떤 순서로 해결할지 계획을 세운다.

4. **Validation (검증력)**
   - 결과가 맞는지 확인하고, 오류가 있으면 재시도한다.
   - 에이전트의 신뢰도를 높이는 핵심 요소다.

### 한 줄 정리

- Agentic Analytics는 SQL을 한 번 생성하는 수준을 넘어서, **질문 분해 -> 실행 -> 검증 -> 재시도**를 반복하는 분석 방식이다.

### 한 줄 비유로 친해지기

- **LangGraph** = "에이전트가 어떤 순서로 일할지"를 그림(흐름도)으로 그리는 도구. 신호등처럼 상태에 따라 다음 단계를 고른다.
- **StateGraph** = 그 흐름도의 설계도. "질문 -> SQL 생성 -> 실행 -> (에러면 재시도) -> 답변" 같은 길을 그린다.
- **TypedDict** = 에이전트가 머릿속에 적어두는 메모지 양식. 질문, SQL, 결과, 에러 칸을 정리해 두는 틀이다.

### SQL 에이전트 아키텍처

20H에서는 위 개념을 실제 SQL 에이전트 구조로 묶는다.

핵심 아키텍처는 보통 이 4개 노드다.

1. **generate_sql**
   - 질문과 스키마, 비즈니스 규칙을 보고 SQL을 만든다.

2. **run_sql**
   - 생성된 SQL을 실제 DB에 실행한다.

3. **validate**
   - SQL 오류나 이상한 결과를 확인한다.

4. **answer**
   - 최종 결과를 자연어 답변으로 바꾼다.

즉, 에이전트는 단순히 SQL만 만드는 것이 아니라:

- 질문 이해
- SQL 생성
- SQL 실행
- 결과 검증
- 최종 답변

까지를 하나의 흐름으로 묶는다.

### 에이전트의 실제 동작 흐름

강의 기준 핵심 흐름은 이렇다.

1. 사용자가 질문한다
2. 에이전트가 SQL을 만든다
3. SQL을 실행한다
4. 오류가 없으면 답변을 만든다
5. 오류가 있으면 에러 메시지를 보고 SQL을 다시 고친다
6. 최대 횟수 안에서 재시도한다

이 점에서 Agentic Analytics는 단순 Text-to-SQL보다 훨씬 실무형이다.

> 핵심 포인트: 좋은 에이전트는 "첫 SQL이 맞는가"보다, **틀렸을 때 스스로 고칠 수 있는가**가 더 중요하다.

### 전반적인 설계 포인트

- **스키마 수집**: DB 구조를 먼저 모아 LLM이 참고하게 한다.
- **비즈니스 규칙 주입**: 예를 들어 `"completed만 유효"` 같은 규칙을 프롬프트에 넣는다.
- **보안 가드레일**: 위험한 SQL 명령은 막고, 허용 테이블만 접근하게 한다.
- **재시도 패턴**: 에러가 나면 그대로 끝내지 않고 수정 후 다시 실행한다.
- **정답률 측정**: 도메인 질문 10개 같은 평가셋으로 실제 성능을 확인한다.

### 한 줄 정리 추가

- Agentic Analytics의 구현체는 보통 **스키마 + 규칙 + SQL 생성 + 실행 + 검증 + 재시도**로 이루어진 SQL 에이전트다.
