## Advanced RAG Query And Retrieval

### Source

- [AI 기반 SQL 분석 에이전트 구축 - 17. Advanced RAG 쿼리변환](https://siapapa.github.io/day3/17-advanced-rag-query/)
- [AI 기반 SQL 분석 에이전트 구축 - 18. Advanced RAG 검색고도화](https://siapapa.github.io/day3/18-advanced-rag-retrieval/)

### 1. 왜 Advanced RAG가 필요한가

- 기본 RAG는 질문을 그대로 임베딩해서 비슷한 문서를 찾는 방식이라 단순하지만 한계가 있다.
- 질문이 추상적이거나, 복합적이거나, 고유명사와 의미 검색이 섞이면 검색 품질이 쉽게 흔들린다.
- 그래서 고도화는 크게 두 방향으로 간다.
  - **질문을 더 잘 바꾸는 방법**
  - **검색 결과를 더 잘 뽑고 다시 정렬하는 방법**

### 2. 3가지 해결법 비교

17H에서 핵심으로 다루는 세 가지는 이렇다.

| 기법 | 한 줄 설명 | 잘 맞는 상황 | 약점 |
| --- | --- | --- | --- |
| **HyDE** | 가상의 답변을 먼저 만든 뒤 그 답변으로 검색 | 추상적 질문, 의미 검색 | LLM 호출 1회 추가 |
| **Multi-Query** | 질문을 3~5개 변형해 각각 검색 후 합침 | 표현이 다양한 질문, 범용 검색 | 검색 횟수와 결과 수 증가 |
| **Query Decomposition** | 복합 질문을 하위 질문으로 쪼개 각각 검색 | 멀티홉/복합 질문 | 통합 로직이 더 필요 |

핵심 감각은 이렇다.

- **HyDE** = "질문보다 답변이 문서와 더 비슷할 수 있다"
- **Multi-Query** = "한 질문을 여러 표현으로 바꿔 재현율을 높인다"
- **Decomposition** = "한 번에 못 찾으면 문제를 쪼갠다"

### 3. 각 기법별 설명

#### HyDE

- 질문에 대한 **가상의 정답 문서**를 LLM이 먼저 작성한다.
- 그 가상 문서를 임베딩해 실제 문서를 검색한다.
- 질문과 문서의 표현이 멀 때 특히 유용하다.

예:

- 질문: `"야간에 응급 진료를 받으려면 어떻게 하나요?"`
- 문서에는 `"24시간 응급실 운영"`처럼 답변형 표현이 있을 수 있다.

#### Multi-Query

- 원 질문을 여러 관점의 질문으로 변형한다.
- 각 질문으로 검색한 뒤 결과를 합집합으로 모은다.
- 실무에서는 가장 **범용적이고 실용적인 고도화**로 소개된다.

예:

- `"호흡기 관련 진료"`
- `"폐 질환"`
- `"숨이 차는 증상"`

처럼 변형해 더 다양한 문서를 건질 수 있다.

#### Query Decomposition

- 복합 질문을 2~4개의 단순한 하위 질문으로 나눈다.
- 각 하위 질문 결과를 모아서 다시 답변을 만든다.
- 단일 검색으로는 놓치기 쉬운 **멀티홉 질문**에 강하다.

예:

- `"가장 경험이 많은 의사는 누구이고, 그 의사의 야간 진료 가능 여부는?"`
- `"가장 경험이 많은 의사는 누구인가?"`
- `"그 의사의 야간 진료 가능 여부는?"`

### 4. 기법별 특성 정리

| 특성 | Naive | HyDE | Multi-Query | Decomposition |
| --- | --- | --- | --- | --- |
| LLM 호출 | 없음 | 1회 | 1회 | 1회 |
| 검색 횟수 | 1회 | 1회 | 3~5회 | 2~4회 |
| 재현율 | 낮음 | 중간 | 높음 | 질문에 따라 높음 |
| 정밀도 | 기본 | 중간 | 결과 수가 많아 후처리 필요 | 통합 품질에 의존 |
| 어휘 불일치 대응 | 약함 | 강함 | 중간 | 중간 |
| 복합 질문 대응 | 약함 | 약함 | 약함 | 강함 |
| 실무 적합도 | 기본 | 중간 | 높음 | 상황별 |

> 한 줄 정리: **범용성은 Multi-Query, 추상 질문은 HyDE, 복합 질문은 Decomposition**이 강하다.

### 5. BM25 vs 벡터 검색

18H에서 검색 자체를 고도화할 때는 **BM25와 벡터 검색을 같이 비교**한다.

| 방식 | 강점 | 약점 | 잘 맞는 질문 |
| --- | --- | --- | --- |
| **BM25** | 정확한 키워드, 고유명사, 코드/약어에 강함 | 의미 유사어에 약함 | `"김철수 의사"`, `"ICD-J45"` |
| **벡터 검색** | 의미 유사성, 유의어, 추상 질문에 강함 | 정확한 문자열 매칭은 약할 수 있음 | `"호흡기 관련"`, `"밤에 심장 문제"` |

핵심 감각은 이렇다.

- **BM25** = 키워드 정밀도
- **벡터 검색** = 의미 재현율

### 6. 하이브리드 + Re-rank

18H의 결론은 보통 **둘 중 하나를 고르는 게 아니라 조합하는 것**이다.

#### 하이브리드 검색

- BM25 결과와 벡터 검색 결과를 함께 합친다.
- `EnsembleRetriever`처럼 weight를 줘서 비율을 조정한다.
- 예: `weights=[0.4, 0.6]`

도메인 감각:

- 고유명사가 많으면 BM25 비중을 높인다
- 의미 질문이 많으면 벡터 비중을 높인다

#### Re-rank

- 1차 검색 결과를 많이 뽑은 뒤
- CrossEncoder 같은 모델로 다시 정렬한다
- 즉, **후보는 넓게 뽑고, 순서는 정밀하게 다시 세운다**

이 흐름은 보통 이렇게 이해하면 쉽다.

1. BM25 검색
2. 벡터 검색
3. 결과 합치기
4. Re-rank로 Top-k 재정렬
5. 최종 컨텍스트만 LLM에 전달

### 7. 하이브리드 + Re-rank RAG 체인

LCEL 흐름으로 보면:

- `retriever` 단계가 단순 벡터 검색이 아니라
- **하이브리드 검색 + Re-rank 검색기**로 바뀐 형태다

즉 최종 체인은 여전히:

- `retriever | prompt | llm | parser`

인데, 여기서 `retriever`가 더 똑똑해진다.

> 한 줄 정리: RAG 체인의 나머지 구조는 비슷하고, **검색기 품질을 올리는 것**이 핵심이다.

### 8. 전체 파이프라인 비교

18H 기준 전체 파이프라인은 이렇게 단계적으로 좋아진다.

| 단계 | 설명 | 장점 | 비용/속도 |
| --- | --- | --- | --- |
| **벡터만** | 질문 임베딩 후 검색 | 가장 단순 | 가장 저렴 |
| **BM25만** | 키워드 기반 검색 | 고유명사에 강함 | 저렴 |
| **하이브리드** | BM25 + 벡터 결합 | 정밀도와 재현율 균형 | 중간 |
| **하이브리드 + Re-rank** | 후보를 다시 정렬 | 가장 품질이 좋음 | 가장 느리고 비쌈 |

그리고 17H의 쿼리 변환까지 함께 넣으면 실전 그림은 더 넓어진다.

| 층위 | 대표 기법 | 역할 |
| --- | --- | --- |
| 질문 변환 | HyDE, Multi-Query, Decomposition | 검색 전에 질문 자체를 개선 |
| 검색 결합 | BM25 + Vector | 서로 다른 검색 강점 결합 |
| 후정렬 | Re-rank | 후보 중 가장 관련성 높은 문서 재배치 |
| 답변 생성 | RAG Chain | 최종 문맥으로 답변 생성 |

### 9. 실무용 선택 가이드

- **가볍게 시작**: 벡터 검색
- **고유명사/코드가 많다**: BM25 또는 하이브리드
- **질문 표현이 다양하다**: Multi-Query
- **질문이 추상적이다**: HyDE
- **질문이 복합적이다**: Decomposition
- **Top-k 품질이 아쉽다**: Re-rank 추가

실무에서는 보통 이 조합이 가장 자연스럽다.

- **Multi-Query + Re-rank**
- **HyDE + 하이브리드 검색**
- **Decomposition + Multi-Query**

### 한 줄 정리

- 17H는 **질문을 더 잘 바꾸는 방법**, 18H는 **검색 결과를 더 잘 고르는 방법**이다.
- 기본 RAG를 고도화할 때는 보통 **Multi-Query -> 하이브리드 검색 -> Re-rank** 순서로 실용성이 높아진다.
