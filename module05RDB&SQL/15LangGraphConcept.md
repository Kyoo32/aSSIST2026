## LangGraph Concept

### Source

- [AI 기반 SQL 분석 에이전트 구축 - 19. LangGraph 개념](https://siapapa.github.io/day3/19-langgraph-concept/)

### 1. LCEL vs LangGraph 비교

둘 다 체인을 조립한다는 점은 같지만, 쓰임새가 다르다.

| 항목 | LCEL | LangGraph |
| --- | --- | --- |
| 핵심 비유 | 레고 블록 조립 | 상태가 있는 흐름도 |
| 잘하는 일 | 직선형 파이프라인 | 분기, 루프, 재시도, 멀티턴 |
| 흐름 구조 | `prompt | llm | parser` 같은 선형 체인 | 노드와 엣지가 있는 그래프 |
| 상태 관리 | 제한적 | 상태를 중심으로 실행 |
| 적합한 예 | 단순 RAG 체인 | 에이전트, SQL 검증 루프, 멀티턴 챗봇 |

한 줄로 보면:

- **LCEL** = "순서대로 흘러가는 체인"
- **LangGraph** = "조건에 따라 갈라지고 되돌아오는 에이전트 흐름"

### 2. LangGraph 핵심 개념

강의의 핵심 구성요소는 네 가지다.

#### State

- 그래프가 들고 다니는 **공용 메모장**
- 보통 `TypedDict`로 정의한다
- 예: `question`, `sql`, `result`, `error`, `attempts`

#### Node

- 실제 일을 수행하는 함수
- 상태를 입력받아, 바뀐 상태 일부를 반환한다
- 예: SQL 생성 노드, 실행 노드, 검증 노드

#### Edge

- 노드 사이의 연결선
- 어떤 노드 다음에 어디로 갈지 정한다

#### StateGraph

- 위 요소들을 묶어 전체 흐름을 설계하는 객체
- 결국 LangGraph는 `상태 + 노드 + 연결`을 코드로 그리는 방식이다

> 한 줄 정리: LangGraph는 "상태를 들고 노드들을 오가며 일하는 흐름도"다.

### 3. 왜 LangGraph가 필요한가

LCEL만으로는 이런 패턴이 불편해진다.

- 실패하면 다시 시도하기
- 조건에 따라 다른 단계로 분기하기
- 멀티턴 대화에서 이전 상태 이어가기
- 중간 실행 과정을 추적하기

즉, 단순 체인을 넘어서 **에이전트처럼 움직이기 시작하면** LangGraph가 더 잘 맞는다.

### 4. 조건부 분기 + 루프

LangGraph의 대표 기능은 `add_conditional_edges`다.

- 어떤 노드 결과를 보고
- 다음 노드를 다르게 선택할 수 있다

이게 중요한 이유는 **성공이면 종료, 실패면 재시도** 같은 흐름을 자연스럽게 만들 수 있기 때문이다.

예를 들면:

1. `try_task`
2. 성공이면 `success`
3. 실패면 다시 `try_task`

이 구조가 바로 **조건부 분기 + 루프**다.

### 5. 재시도 패턴

강의에서 보여주는 대표 패턴은 이렇다.

- 상태에 `attempts`와 `error`를 넣는다
- 작업 노드가 실패하면 `error`를 기록한다
- 조건 함수가
  - 성공이면 종료
  - 실패이고 시도 횟수가 남아 있으면 재시도
  - 실패이고 한도를 넘으면 종료
  로 분기한다

이 패턴의 장점:

- 실패를 숨기지 않는다
- 몇 번 시도했는지 추적 가능하다
- SQL 에이전트처럼 "생성 -> 실행 -> 실패 -> 수정 -> 재실행" 흐름에 바로 쓸 수 있다

### 6. 조건부 분기 + 루프 (재시도 패턴) 정리

| 요소 | 역할 |
| --- | --- |
| `attempts` | 현재 몇 번 시도했는지 기록 |
| `error` | 실패 원인 저장 |
| 조건 함수 | `success` / `retry` / `stop` 같은 경로 결정 |
| 루프 엣지 | 실패 시 같은 노드나 이전 노드로 되돌림 |

한 줄 감각은 이렇다.

- **에이전트가 한 번에 못 풀면, 다시 시도하게 만드는 구조**

### 7. 실행 추적 `stream()`

LangGraph는 `invoke()`만 있는 게 아니라 `stream()`으로 중간 과정을 볼 수 있다.

- `invoke()` = 최종 결과만 본다
- `stream()` = 어떤 노드를 거쳤는지 단계별로 본다

그래서 디버깅할 때 특히 좋다.

- 어느 노드에서 에러가 났는지
- 몇 번 재시도했는지
- 상태가 어떻게 바뀌는지

를 중간중간 확인할 수 있다.

### 8. (심화) Checkpointer

기본 그래프는 `invoke()`가 끝나면 상태가 사라진다.

그런데 멀티턴 챗봇이나 장시간 실행 에이전트는 **이전 상태를 다시 불러와야** 한다.

이때 붙이는 것이 `Checkpointer`다.

대표 개념:

- `MemorySaver`
  - 메모리 기반 저장
  - 학습/실습용

- `thread_id`
  - 대화 세션 구분자
  - 같은 `thread_id`면 이전 상태를 이어받는다

즉, Checkpointer는:

- **멀티턴 대화 유지**
- **서버 재시작 후 복구**
- **중단된 실행 재개**

에 필요한 장치다.

### 9. Checkpointer가 필요한 순간

| 상황 | 이유 |
| --- | --- |
| 멀티턴 챗봇 | 이전 질문/답변 문맥을 이어야 함 |
| 재시작 대응 | 서버나 프로세스가 끊겨도 상태를 복구해야 함 |
| 장시간 에이전트 | 중간 진행 상황을 저장해야 함 |
| 디버깅/관찰 | 어떤 경로로 실행됐는지 기록 가능 |

강의의 감각으로 보면:

- `ChatState`를 직접 들고 가는 단순 방식보다
- LangGraph + Checkpointer가 더 **에이전트 프레임워크다운 방식**이다

### 10. 실무용 선택 가이드

- **직선형 체인만 필요**: LCEL
- **분기와 루프가 필요**: LangGraph
- **멀티턴/재시작 복구도 필요**: LangGraph + Checkpointer

특히 SQL 에이전트 쪽에서는 LangGraph가 잘 맞는다.

- SQL 생성
- 실행
- 오류 확인
- 수정
- 재시도
- 최종 답변

이 흐름 자체가 그래프 구조이기 때문이다.

### 한 줄 정리

- `LCEL`은 체인을 예쁘게 조립하는 도구이고,
- `LangGraph`는 **조건 분기, 루프, 재시도, 상태 지속**이 필요한 에이전트를 만드는 도구다.
