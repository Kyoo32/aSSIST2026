## Vanna Intro

### Source

- [AI 기반 SQL 분석 에이전트 구축 - 13. Vanna Intro](https://siapapa.github.io/day3/13-vanna-intro/)

### 1. LlamaIndex vs Vanna 비교

강의의 한 줄 비교는 이렇다.

- **LlamaIndex** = 전체 레시피북을 매번 펼쳐주는 방식
- **Vanna** = 학습하는 요리사

구조적으로 보면:

- **LlamaIndex**
  - 질문 -> 스키마 전체를 프롬프트에 주입 -> LLM -> SQL

- **Vanna**
  - 질문 -> 유사한 DDL / 문서 / SQL 쌍 검색 -> 검색 결과를 프롬프트에 주입 -> LLM -> SQL

즉, LlamaIndex는 프롬프트 중심이고, Vanna는 검색 기반 학습 자산 활용이 중심이다.

### 2. 상세 비교표

| 특성 | LlamaIndex | Vanna |
| --- | --- | --- |
| 프롬프트 구성 | 전체 스키마 고정 주입 | 질문에 관련된 것만 검색/주입 |
| 테이블 30개 이상 | 토큰 초과, 정확도 하락 가능 | 관련 테이블만 검색해 확장 가능 |
| 학습 | few-shot 수동 추가 | DDL / 문서 / SQL 쌍 누적 학습 |
| 정확도 개선 | 프롬프트 튜닝 중심 | 학습 자산 추가로 자동 개선 |
| 설정 난이도 | 비교적 간단 | 초기 학습이 필요해 조금 더 복잡 |
| 대규모 DB 적합도 | 낮은 편 | 높은 편 |
| 비용 효율 | 매 질문마다 전체 스키마 토큰 사용 | 관련 부분만 사용 |

> 한 줄 정리: DB가 작고 단순하면 LlamaIndex도 충분하지만, DB가 커지고 반복 질문이 많아질수록 Vanna 쪽이 유리해진다.

### 3. Vanna 내부 RAG 검색 흐름

Vanna는 질문을 받으면 내부적으로 3개의 벡터 컬렉션을 검색한다.

1. **DDL 컬렉션 검색**
   - 관련 테이블 구조를 찾는다

2. **Documentation 컬렉션 검색**
   - 비즈니스 규칙, 용어 설명을 찾는다

3. **SQL Pairs 컬렉션 검색**
   - 비슷한 질문과 정답 SQL 예시를 찾는다

그다음:

- 검색된 DDL + 문서 + SQL 예시를 합쳐서 컨텍스트를 만들고
- 그 컨텍스트와 질문을 함께 LLM에 넣어 SQL을 생성한다

즉, Vanna는 단순히 "스키마를 보여주는" 방식이 아니라,
**관련된 학습 자산을 먼저 찾아서 꺼내 쓰는 Text-to-SQL용 RAG**라고 볼 수 있다.

### 4. Vanna가 학습하는 데이터는 3가지 + 학습 자산 우선순위

Vanna가 학습하는 자산은 3가지다.

| 종류 | 설명 | 예시 | 비유 |
| --- | --- | --- | --- |
| **DDL** | 테이블 구조 정보 | `CREATE TABLE patients (...)` | 재료 목록 |
| **Documentation** | 비즈니스 용어 / 규칙 | `"completed만 유효한 진료"` | 조리법 메모 |
| **SQL Pairs** | 질문-정답 SQL 쌍 | `("환자 수?", "SELECT COUNT(*) ...")` | 예전 요리 사진 |

학습 자산 우선순위는 강의에서 분명하다.

1. **DDL 먼저**
   - 테이블 구조를 모르면 SQL 생성 자체가 어렵다

2. **Documentation 다음**
   - 비즈니스 규칙과 용어를 알려준다

3. **SQL Pairs 마지막**
   - 정답 예시를 통해 패턴을 더 정확히 학습시킨다

> 핵심 포인트: 실무에서는 DDL이 기반이고, 문서가 의미를 보강하고, SQL 예시가 정확도를 끌어올린다.

### 한 줄 정리

- Vanna는 SQL 생성에 특화된 RAG 접근으로, **DDL -> Documentation -> SQL Pairs**를 축적해 가며 Text-to-SQL 정확도를 점진적으로 높이는 방식이다.
