## LangSmith And Ragas Monitoring

### Source

- [AI 기반 SQL 분석 에이전트 구축 - 21. LangSmith](https://siapapa.github.io/day4/21-langsmith/)

### 1. 왜 모니터링이 필요한가

일반 소프트웨어는 입력과 로직이 비교적 고정적이지만, LLM 애플리케이션은 실행 결과가 매번 조금씩 달라질 수 있다.

그래서 단순히 `print()` 몇 줄로는 부족하다.

- 어떤 프롬프트가 들어갔는지
- 어떤 도구가 호출됐는지
- 중간 단계에서 어디서 느려졌는지
- 비용과 토큰이 얼마나 들었는지
- 실패가 검색 문제인지, 생성 문제인지

를 함께 봐야 한다.

한 줄로 보면:

- **모니터링은 "에이전트가 실제로 어떻게 일했는지"를 관찰하는 장치**다.

### 2. Run / Trace 구조

LangSmith의 기본 단위는 `Run`과 `Trace`다.

- **Trace**
  - 한 번의 전체 실행
  - 예: 사용자의 질문 하나가 끝까지 처리된 전체 흐름

- **Run**
  - Trace 안의 개별 작업 단위
  - 예: LLM 호출, Retriever 실행, Tool 호출, Parser 단계

즉:

- **Trace = 전체 여정**
- **Run = 여정 안의 한 단계**

에이전트나 RAG 체인은 보통 여러 Run이 트리 구조로 묶여 하나의 Trace를 만든다.

### 3. LangSmith UI에서 볼 수 있는 것

LangSmith UI에서는 대체로 아래를 확인한다.

- **Trace 목록**
  - 어떤 요청이 성공했고 무엇이 실패했는지 빠르게 훑어본다

- **Trace 상세**
  - 한 요청의 전체 흐름을 시간 순서대로 본다

- **Run Tree**
  - 어떤 노드가 어떤 하위 노드를 불렀는지 계층적으로 본다

- **입력 / 출력**
  - 프롬프트, 검색 결과, 모델 응답, 최종 답변을 단계별로 확인한다

- **토큰 / 비용 / 지연시간**
  - 어느 단계가 비싸고 느린지 파악한다

- **Feedback / Annotation**
  - 사람이 좋은 결과와 나쁜 결과를 표시해 품질 개선에 활용한다

- **Dataset / Experiment**
  - 고정 질문 세트로 반복 평가하고 버전 간 비교를 한다

### 4. LangSmith UI에서 특히 중요한 해석 포인트

강의 흐름에서 중요한 것은 "보는 것"보다 "왜 봐야 하는가"다.

- 답변이 틀렸다면:
  - 검색 문서가 잘못됐는지
  - 프롬프트가 애매했는지
  - 모델이 문서를 잘못 해석했는지

- 응답이 느리다면:
  - 검색이 병목인지
  - LLM 호출이 병목인지
  - 재시도 루프가 과도한지

- 비용이 크다면:
  - 불필요하게 긴 컨텍스트를 넣었는지
  - 호출 횟수가 너무 많은지

즉, LangSmith는 단순 로그 뷰어가 아니라 **병목과 실패 원인을 분해해서 보는 디버깅 도구**다.

### 5. 4대 메트릭

Ragas에서 핵심적으로 보는 대표 지표는 네 가지다.

- **Faithfulness**
  - 답변이 주어진 문맥에 충실한가
  - 문맥에 없는 내용을 지어내지 않았는가

- **Answer Relevancy**
  - 답변이 질문에 제대로 맞는가
  - 엉뚱한 방향으로 벗어나지 않았는가

- **Context Precision**
  - 검색된 문맥 중 실제로 필요한 문서가 얼마나 많이 포함됐는가
  - 불필요한 문서가 너무 많이 섞이지 않았는가

- **Context Recall**
  - 답변에 필요한 핵심 문맥을 검색이 놓치지 않았는가
  - 중요한 문서를 빠뜨리지 않았는가

### 6. 4대 메트릭 요약

| 메트릭 | 무엇을 보는가 | 낮게 나오면 의심할 것 |
| --- | --- | --- |
| Faithfulness | 답변이 근거 문맥을 벗어나지 않는가 | 환각, 과장, 근거 없는 생성 |
| Answer Relevancy | 답변이 질문 의도에 맞는가 | 질문 해석 실패, 초점 이탈 |
| Context Precision | 검색 결과가 얼마나 깔끔하게 관련 문서 중심인가 | 잡음 문서 과다, 검색 정밀도 부족 |
| Context Recall | 필요한 문서를 빠뜨리지 않았는가 | 검색 누락, 재현율 부족 |

감각적으로 보면:

- **Faithfulness** = 답변이 근거를 지켰는가
- **Answer Relevancy** = 답변이 질문을 제대로 받았는가
- **Precision** = 검색 결과가 쓸모 있는 것 위주인가
- **Recall** = 필요한 근거를 빠뜨리지 않았는가

### 7. 모니터링과 평가를 같이 봐야 하는 이유

LangSmith와 Ragas는 역할이 다르다.

- **LangSmith**
  - 실행 과정을 본다
  - 어디서 실패했는지 추적한다

- **Ragas**
  - 결과 품질을 점수로 본다
  - 검색과 답변의 품질을 정량 비교한다

둘을 같이 써야:

- "점수가 왜 낮은지"를 Trace로 추적할 수 있고
- "개선이 실제로 효과 있었는지"를 메트릭으로 다시 확인할 수 있다

### 8. 전체 흐름으로 정리

실무 감각으로 정리하면 순서는 이렇다.

1. LangSmith로 Trace를 보며 실패 지점을 찾는다.
2. 검색 문제인지, 프롬프트 문제인지, 생성 문제인지 나눈다.
3. 체인이나 프롬프트를 수정한다.
4. Ragas 메트릭으로 전후 성능을 비교한다.

즉:

- **LangSmith = 관찰**
- **Ragas = 평가**
- **개선 사이클 = 추적 -> 수정 -> 재평가**

### 한 줄 정리

- LangSmith는 에이전트 실행을 보이게 만드는 관측 도구이고, Ragas는 검색과 답변 품질을 4대 메트릭으로 점검하는 평가 도구다.
