## LCEL RAG Chain

### Source

- [AI 기반 SQL 분석 에이전트 구축 - 16. LCEL RAG Chain](https://siapapa.github.io/day3/16-lcel-rag-chain/)

### 1. LCEL RAG 체인 조립

강의의 핵심 패턴은 이 한 줄로 요약된다.

> **RAG 체인의 핵심 패턴: 검색기 | 포맷 + 프롬프트 | LLM | 파서**

흐름을 풀어 쓰면 다음과 같다.

1. **검색기 (`retriever`)**
   - 질문과 관련된 문서를 찾는다.

2. **포맷 (`format_docs`)**
   - 검색된 문서 리스트를 하나의 텍스트로 묶는다.

3. **프롬프트 (`rag_prompt`)**
   - 질문과 컨텍스트를 합쳐 LLM 입력을 만든다.

4. **LLM**
   - 컨텍스트 기반 답변을 생성한다.

5. **파서 (`StrOutputParser`)**
   - 모델 출력을 최종 문자열 형태로 정리한다.

LCEL에서는 이걸 파이프처럼 연결한다.

- `retriever | format_docs`
- `| rag_prompt`
- `| llm`
- `| StrOutputParser()`

> 한 줄 정리: RAG 체인은 "찾고, 묶고, 물어보고, 답을 정리하는" 조립식 파이프라인이다.

### 2. 스트리밍이 중요한 이유

- LLM 응답은 보통 몇 초가 걸린다.
- 사용자는 아무 반응이 없으면 느리다고 느끼기 쉽다.

스트리밍을 쓰면:

- 첫 토큰이 빨리 보이고
- 답이 생성되는 과정이 실시간으로 드러나며
- 체감 대기 시간이 크게 줄어든다

강의 기준 핵심 포인트:

- 전체 답변이 다 끝날 때까지 기다리는 것보다
- **0.5초 안에 첫 글자라도 보이는 경험**이 훨씬 낫다

그래서 Gradio 같은 채팅 UI에서도 스트리밍이 중요하다.

> 한 줄 정리: 스트리밍은 속도를 실제로 높이는 것보다, 사용자가 "덜 기다리게 느끼도록" 만드는 UX 장치다.

### 3. Fallback 사용 시 주의

- `with_fallbacks()`는 기본 모델이 실패했을 때 대체 모델로 넘어가게 하는 방식이다.
- 예:
  - API 키 오류
  - 요금 한도 초과
  - 모델 과부하

중요한 주의점:

> **Fallback은 예외(Exception)가 발생했을 때만 동작한다.**

즉,

- 응답은 정상적으로 왔지만 품질이 낮은 경우
- 답이 틀렸지만 에러는 아닌 경우

에는 fallback이 작동하지 않는다.

그래서 fallback은 "품질 개선 장치"라기보다 **가용성 보강 장치**에 가깝다.

### 4. RAG 체인 구성요소 정리

강의의 전체 데이터 흐름은 이렇게 볼 수 있다.

1. **사용자 질문**
   - 예: `"내과에 어떤 의사가 있나요?"`

2. **검색 (`retriever`)**
   - 질문 임베딩
   - 벡터 DB에서 관련 문서 Top-K 검색

3. **포맷 (`format_docs`)**
   - 검색 결과를 하나의 컨텍스트 문자열로 합침

4. **프롬프트 조립 (`rag_prompt`)**
   - 컨텍스트 + 질문을 LLM 입력으로 구성

5. **LLM 호출**
   - 문맥 기반 답변 생성

6. **파싱**
   - 최종 출력 형태 정리

추가 팁:

- `format_docs`는 단순 줄바꿈 결합만 하는 것이 아니라,
- 문서마다 번호를 붙이는 식으로 커스터마이즈할 수도 있다.

### 한 줄 정리

- LCEL 기반 RAG 체인은 결국 **검색 -> 포맷 -> 프롬프트 -> LLM -> 파서**의 흐름을 얼마나 명확하게 조립하느냐의 문제다.
