## Text To SQL Improvement Strategies

### Source

- [AI 기반 SQL 분석 에이전트 구축 - 10. Text-to-SQL Advanced](https://siapapa.github.io/day2/10-text-to-sql-advanced/)

### 개선 전략 3가지

강의에서 제시한 개선 전략은 다음 3가지다.

1. **table_info 보강**
   - 스키마에 자연어 설명을 더한다

2. **few-shot 주입**
   - 질문- SQL 예시 쌍으로 패턴을 보여준다

3. **테이블 선택 자동화**
   - `ObjectIndex`로 관련 테이블만 먼저 고른다

### 전략 1. 도메인 지식 주입 (`context_str_prefix`)

- 이 전략은 스키마만 보여주는 대신, **도메인 규칙과 비즈니스 맥락**을 함께 프롬프트에 넣는 방식이다.
- 예를 들면 이런 정보들이다.
  - `visits.status='completed'`만 집계 대상
  - `visits.cost`는 원(KRW) 단위
  - `"지난달"`의 날짜 계산 패턴
  - `"재방문"`의 정의

왜 중요한가:

- LLM은 단순 컬럼명만 봐서는 비즈니스 규칙을 알 수 없다.
- 따라서 질문이 조금만 모호해져도 자의적으로 해석하기 쉽다.
- 도메인 지식을 넣어주면 SQL 생성 기준이 더 명확해진다.

> 한 줄 정리: 전략 1은 "스키마만 주지 말고, 이 DB가 실제로 어떻게 해석돼야 하는지도 같이 알려주는 것"이다.

### 전략 2. 커스텀 프롬프트 + Few-shot

- 이 전략은 프롬프트 안에 **질문- SQL 예시 쌍**을 몇 개 넣어, LLM이 원하는 패턴을 따라 하게 만드는 방식이다.
- 예를 들어:
  - `"전체 환자 수는?" -> COUNT(*)`
  - `"내과 의사 목록"` -> `JOIN departments`
  - `"지난달 완료된 진료 건수"` -> `status='completed' + DATE_TRUNC`

왜 중요한가:

- LLM은 설명만 읽는 것보다 **예시를 보고 패턴을 모방**할 때 더 안정적으로 동작하는 경우가 많다.
- 특히 자주 쓰는 JOIN 경로, 날짜 조건, 별칭 스타일 같은 것을 예시로 고정할 수 있다.

언제 유리한가:

- 비슷한 질문이 반복되는 도메인
- 자주 틀리는 쿼리 패턴이 있는 경우
- 한국어 표현을 특정 SQL 스타일로 안정화하고 싶을 때

> 한 줄 정리: 전략 2는 "설명만 하지 말고, 모범답안을 몇 개 보여줘서 따라 하게 만드는 것"이다.

### 전략 3. 테이블 자동 선택 (`ObjectIndex`)

- 실제 DB에 테이블이 많아지면 모든 스키마를 프롬프트에 다 넣는 것은 비효율적이다.
- 이 전략은 질문과 관련된 테이블만 먼저 검색해서, **Top-K 테이블만 Text-to-SQL에 넘기는 방식**이다.

강의에서 설명한 주요 구성요소:

- **`SQLTableSchema`**: 각 테이블에 자연어 설명을 붙인 객체
- **`SQLTableNodeMapping`**: 테이블을 벡터 검색 가능한 노드로 바꾸는 연결부
- **`ObjectIndex`**: 질문과 가장 관련된 테이블을 검색하는 인덱스
- **`SQLTableRetrieverQueryEngine`**: 테이블 검색 + Text-to-SQL을 합친 엔진

왜 중요한가:

- 관련 없는 테이블까지 프롬프트에 넣으면 토큰이 늘고, LLM이 헷갈릴 가능성도 커진다.
- 반대로 관련 테이블만 좁혀주면 더 짧고 정확한 프롬프트를 만들 수 있다.

> 한 줄 정리: 전략 3은 "질문에 필요한 테이블만 먼저 고르고, 그 범위 안에서 SQL을 만들게 하는 것"이다.

### 실전 감각으로 묶어보기

- **전략 1**은 해석 기준을 보강한다.
- **전략 2**는 답변 패턴을 학습시킨다.
- **전략 3**은 문제 범위를 줄인다.

셋을 함께 보면:

- 무엇을 의미하는지 더 잘 알게 하고
- 어떻게 SQL을 써야 하는지 예시를 주고
- 어떤 테이블만 봐야 하는지도 먼저 골라주는 구조다

### 한 줄 정리

- Text-to-SQL 개선의 핵심은 모델을 막연히 더 똑똑하게 만들기보다, **더 좋은 문맥을 주고, 더 좋은 예시를 주고, 더 좁은 범위에서 생각하게 만드는 것**이다.
